# 주문 처리 흐름

이 문서는 로컬 학습/포트폴리오용 쇼핑몰 예제의 주문 저장 흐름을 빠르게 파악하기 위한 요약입니다.

## 1. 상품 상세 선택

- 사용자는 상품 목록 또는 검색 결과에서 상품 상세 화면으로 이동합니다.
- 상세 화면에서는 상품 ID를 기준으로 상품 정보와 옵션을 확인하고, 구매 수량과 옵션을 선택합니다.

## 2. `cart_edit.php` 장바구니 저장

- 상품 상세에서 선택한 상품 ID, 수량, 옵션 정보는 `cart_edit.php`를 통해 장바구니 쿠키에 저장됩니다.
- 장바구니는 쿠키 기반 JSON 데이터로 유지되며, 주문 저장 단계에서 서버가 상품/옵션 정보를 다시 확인합니다.

## 3. `cart.php` 장바구니 조회

- `cart.php`는 장바구니 쿠키를 읽어 담긴 상품을 보여줍니다.
- 화면 표시용 상품명, 가격, 옵션명은 DB 조회 결과를 기준으로 구성됩니다.

## 4. `order.php` 주문자/수령자 입력

- `order.php`에서 주문자와 수령자 정보를 입력합니다.
- 회원 정보가 있으면 주문 입력 편의를 위해 일부 기본값을 사용할 수 있습니다.

## 5. `order_pay.php` 결제 정보 확인

- `order_pay.php`에서 결제 수단과 결제 관련 입력값을 확인합니다.
- 이 프로젝트는 실결제 PG 검증을 구현한 상용 결제 시스템이 아니라 학습용 결제 흐름 예제입니다.

## 6. `order_ok.php` 주문 저장

- `order_ok.php`는 최종 주문 저장 처리 파일입니다.
- 장바구니 쿠키 값을 그대로 신뢰하지 않고 상품 ID, 수량, 옵션 ID를 검증하며 상품 가격은 DB에서 다시 조회합니다.

## 7. `jumun` 주문 마스터와 `jumuns` 주문 상세 분리 이유

- `jumun`은 주문번호, 주문일, 주문자/수령자, 결제/배송 상태, 총액 같은 주문 대표 정보를 저장하는 주문 마스터 역할입니다.
- `jumuns`는 한 주문에 포함된 여러 상품 라인을 저장하는 주문 상세 역할입니다.
- 이 구조는 한 번의 주문에 여러 상품이 들어갈 수 있는 1:N 관계를 표현하기 쉽고, 주문 목록과 주문 상세 조회를 분리하기 좋습니다.

## 8. 상품 가격을 DB에서 재조회하는 이유

- 클라이언트 쿠키나 화면의 가격 값은 사용자가 조작할 수 있습니다.
- 최종 주문 저장 시 DB의 현재 상품 가격을 재조회하면 가격 위변조 위험을 줄이고 주문 금액 계산의 기준을 서버 데이터로 통일할 수 있습니다.

## 9. 트랜잭션 적용 이유

- 주문 저장은 주문 마스터(`jumun`)와 주문 상세(`jumuns`)가 함께 저장되어야 의미가 있습니다.
- 중간 단계에서 오류가 발생하면 일부 데이터만 저장되는 불일치를 막기 위해 트랜잭션으로 commit/rollback을 처리합니다.

## 10. 성공 후 장바구니 쿠키 삭제 흐름

- 주문 저장이 성공하면 장바구니 쿠키를 만료시켜 주문 완료 후 같은 상품이 장바구니에 남지 않도록 합니다.
- 실패 시에는 rollback 후 일반 오류 메시지를 보여주며, 내부 오류는 로그 확인 대상으로 남깁니다.
