# 쇼핑몰 판매관리 사이트

## 1. 프로젝트 소개

이 프로젝트는 PHP와 MySQL 기반으로 구현한 의류 중심 쇼핑몰 판매관리 웹사이트입니다.  
일반 사용자가 상품을 탐색하고, 옵션을 선택해 장바구니에 담고, 주문 및 결제를 진행할 수 있도록 구성했으며, 관리자 페이지에서는 회원, 상품, 주문, 옵션, FAQ를 통합적으로 관리할 수 있도록 구현했습니다.

단순히 상품 목록만 보여주는 정적 화면이 아니라, 실제 쇼핑몰의 기본 흐름인 회원가입, 로그인, 상품 검색, 카테고리 분류, 상세 페이지, 장바구니, 주문서 작성, 결제, 주문 조회, 고객 문의, FAQ, 관리자 관리 기능까지 한 프로젝트 안에 담은 실습형 웹 서비스입니다.

프론트는 Bootstrap을 활용해 반응형 UI를 구성했고, 백엔드는 PHP와 MySQL을 사용해 데이터 조회, 입력, 수정, 삭제가 가능한 구조로 만들었습니다.  
또한 비회원 주문 조회, 옵션 조합 기반 주문 처리, 관리자 전용 주문 상태 변경, FAQ 관리, 상품 옵션 관리 등 실제 운영 흐름을 고려한 기능까지 포함하여 쇼핑몰 운영 구조를 종합적으로 이해하고 구현하는 데 목적을 둔 프로젝트입니다.

---

## 2. 프로젝트 목표

이 프로젝트의 핵심 목표는 다음과 같습니다.

- PHP 기반 서버 사이드 렌더링 방식에 익숙해지고, 쇼핑몰 서비스의 전형적인 흐름을 직접 구현한다.
- MySQL 데이터베이스를 활용해 회원, 상품, 주문, 옵션, 게시판 데이터를 체계적으로 관리한다.
- 관리자 페이지와 사용자 페이지를 분리하여, 판매자와 고객의 역할에 맞는 기능을 각각 구현한다.
- 상품 목록 조회부터 주문 완료까지 이어지는 전체 구매 프로세스를 하나의 흐름으로 완성한다.
- 단순 CRUD를 넘어 실제 서비스에서 자주 필요한 기능인 검색, 페이징, 비회원 주문 조회, 옵션 조합 처리, 주문 상태 관리 등을 구현한다.

## 4. 핵심 구현 포인트

## 4-1. 장바구니를 쿠키 기반으로 구현

장바구니는 DB가 아니라 쿠키에 JSON 형태로 저장되도록 구현했습니다.  
상품 ID, 수량, 옵션 값들을 `^` 구분자로 직렬화한 뒤 배열 형태로 저장하고, 이를 JSON으로 인코딩하여 쿠키에 보관합니다.

이 방식의 장점은 다음과 같습니다.

- 로그인하지 않아도 장바구니 사용 가능
- 세션이나 별도 장바구니 테이블 없이 빠르게 기능 구현 가능
- 상품 옵션 조합을 간단하게 관리 가능

예를 들어 장바구니 한 항목은 다음과 같은 구조로 저장됩니다.

- 상품 ID
- 수량
- 옵션1 값
- 옵션2 값
- 옵션3 값

즉, 단순 상품 담기가 아니라 옵션 조합이 반영된 장바구니 구조라는 점이 특징입니다.

## 4-2. 주문 데이터와 주문 상세 데이터를 분리 저장

주문 정보는 두 개의 테이블로 나누어 관리됩니다.

### 1) jumun
주문 전체를 대표하는 마스터 테이블입니다.

- 주문번호
- 주문자 정보
- 수령자 정보
- 결제 방식
- 총 주문금액
- 주문 상태

### 2) jumuns
주문에 포함된 각 상품을 저장하는 상세 테이블입니다.

- 주문번호
- 상품 ID
- 수량
- 단가
- 할인 적용 금액
- 옵션 값

이렇게 주문 마스터와 주문 상세를 분리함으로써 한 주문에 여러 상품이 담겨도 구조적으로 관리할 수 있도록 설계했습니다.  
쇼핑몰 주문 구조를 관계형 데이터베이스 관점에서 이해하고 구현했다는 점에서 의미가 있습니다.

## 4-3. 주문번호 자동 생성

주문번호는 `YYMMDD + 4자리 순번` 형식으로 생성됩니다.

예시:
- 2506020001

생성 방식은 다음과 같습니다.

1. 현재 날짜 기준 접두어 생성
2. 해당 날짜로 시작하는 기존 주문번호 중 최대값 조회
3. 마지막 순번을 1 증가
4. 4자리로 맞춰 주문번호 완성

이 방식은 사람이 보기에도 직관적이고, 날짜별 주문 흐름을 관리하기에도 적합합니다.

상품 옵션은 고정 문자열로 박아 넣지 않고, 별도의 옵션 테이블 구조로 설계했습니다.

### opt
상위 옵션 테이블
- 색상
- 사이즈
- 길이

### opts
하위 옵션 테이블
- 검정색, 흰색, 네이비 등
- XS, S, M, L, XL 등
- 40cm, 45cm, 50cm 등

이 구조 덕분에 상품마다 옵션을 유연하게 조합할 수 있고, 관리자 페이지에서 옵션 구성을 직접 수정할 수 있도록 만들었습니다.

## 4-5. 비회원 주문 조회 기능

회원뿐 아니라 비회원도 이름과 이메일을 입력하면 주문 조회가 가능하도록 구현했습니다.  
이는 실제 쇼핑몰에서도 자주 필요한 기능이며, 회원 전환 없이도 주문 경험을 제공할 수 있다는 장점이 있습니다.

비회원 조회 흐름은 다음과 같습니다.

1. 이름 + 이메일 입력
2. 해당 정보로 주문 존재 여부 확인
3. 조회 가능 시 관련 쿠키 저장
4. 주문 목록 페이지로 이동
5. 이후 상세 주문 조회 가능

## 4-6. 공통 페이징 함수 구현

여러 페이지에서 반복되는 페이징 처리를 `mypagination()` 함수로 공통화했습니다.

이 함수는 다음 역할을 수행합니다.

- 전체 레코드 수 조회
- 현재 페이지에 해당하는 범위 계산
- limit 적용 SQL 실행
- 페이지바 HTML 생성

적용 페이지 예시:
- 상품 목록
- 상품 검색
- Q&A 게시판
- 관리자 상품 목록
- 관리자 주문 목록
- 관리자 회원 목록
- 옵션 목록

공통 함수로 분리함으로써 중복 코드를 줄이고 재사용성을 높였습니다.

9. 실행 방법
9-1. 데이터베이스 생성

먼저 MySQL 또는 MariaDB에서 데이터베이스를 생성합니다.

CREATE DATABASE shop62 DEFAULT CHARACTER SET utf8mb4;

그 다음 db/shop62.sql 파일을 import 합니다.

예시:

mysql -u root -p shop62 < db/shop62.sql
9-2. DB 연결 설정

common.php에서 DB 연결 정보를 확인합니다.

$db = mysqli_connect("localhost", "root", "", "shop62");

로컬 환경에 맞게 아래 값을 수정하면 됩니다.

host
user
password
database name
9-3. 웹 서버 실행

Apache + PHP + MySQL이 실행 가능한 로컬 환경에서 프로젝트 폴더를 웹 루트에 배치한 후 실행합니다.

예시:

XAMPP
Laragon
Apache + PHP 수동 구성 환경

브라우저 접속 예시:

http://localhost/shopping/main.php

또는 환경에 따라

http://localhost/main.php
10. 관리자 계정

프로젝트 내부 설정 기준 관리자 계정은 다음과 같이 정의되어 있습니다.

아이디: admin
비밀번호: 1234

해당 값은 common.php에 하드코딩되어 있으므로 실제 배포 환경에서는 반드시 변경이 필요합니다.

11. 구현하면서 중점적으로 신경 쓴 부분
1) 실제 쇼핑몰 흐름을 한 번에 경험할 수 있도록 구성

회원 기능, 상품 기능, 주문 기능, 관리자 기능을 각각 따로 만들지 않고 하나의 연결된 서비스 흐름으로 구현했습니다.

2) 사용자와 관리자 관점을 분리

사용자는 상품 탐색과 구매에 집중하고, 관리자는 운영과 데이터 관리에 집중할 수 있도록 페이지 구조를 분리했습니다.

3) 옵션 기반 상품 구매 흐름 구현

상품마다 여러 옵션을 선택할 수 있도록 하여 단순 상품 구매보다 더 현실적인 쇼핑몰 구조를 만들었습니다.

4) 주문 데이터 정규화

주문 전체 정보와 주문 상품 상세를 분리 저장함으로써 데이터 구조를 더 명확하게 설계했습니다.

5) 반복 기능 공통화

페이징과 공통 레이아웃 등 반복되는 부분을 함수 및 include 구조로 묶어 유지보수를 쉽게 했습니다.

12. 아쉬운 점 및 개선하고 싶은 부분

이 프로젝트는 쇼핑몰 핵심 흐름을 구현하는 데 초점을 맞춘 만큼, 향후 아래와 같은 개선이 가능합니다.
